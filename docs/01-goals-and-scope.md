# 01 Goals And Scope

## 背景

OpenCode 已经具备完整的会话管理、消息处理、工具调用、权限询问、问题回问和事件流能力。飞书 IMUI 的核心不是重新实现一套 agent 系统，而是把飞书的消息交互协议映射到 OpenCode 的现有模型上。

飞书版本作为第一站，有三个现实优势：

- 企业内协作入口天然集中在 IM
- 飞书提供机器人、事件订阅、消息卡片和长连接接入机制
- 用户对“异步执行 + 状态回推 + IM 内完成审批”的心智比较成熟

本项目的一个重要前提已经明确：

- 部署机器处于 NAT 网络下，没有公网 IP
- 机器可以正常访问互联网

因此本方案默认不走公网 webhook，而是基于飞书长连接能力完成消息接入和卡片交互。

## 项目目标

### 用户目标

- 用户可以在飞书中通过 `@机器人` 发送自然语言请求
- 用户可以在同一会话中连续追问，保持上下文
- 用户可以看到 OpenCode 当前是否在执行、执行到什么阶段、是否需要审批
- 用户可以在飞书中完成权限审批和问题回答
- 用户可以收到最终结果和必要的补充信息

### 工程目标

- 复用 OpenCode 的 HTTP API、SSE、Session、Permission、Question 机制
- IM 接入实现为独立服务，降低对 OpenCode 主仓的侵入
- 把飞书协议适配和 IM 抽象分层，为后续扩展 Slack、企业微信、Discord 预留空间
- 设计出便于灰度和排障的日志、追踪和状态记录结构
- 在无公网入口的前提下完成接入，只依赖公网出站能力

## 非目标

- 首版不做多 IM 平台统一运营后台
- 首版不做复杂的飞书富文本编辑器体验
- 首版不做文件上传后的完整多模态理解闭环
- 首版不做跨多个代码仓的单条消息编排
- 首版不把 OpenCode Web 完整嵌入飞书
- 首版不兼容飞书旧版消息卡片回传协议
- 首版不设计公网 webhook 入口

## MVP 范围

### 必做

- 飞书机器人接收私聊消息和群聊 `@`
- 飞书长连接接收消息事件
- 飞书长连接接收新版卡片交互
- 为会话建立 `chat/thread -> sessionID` 映射
- 支持新建会话、连续对话、取消执行、查看状态
- 支持把 OpenCode 结果回推成飞书消息
- 支持权限审批
- 支持问题回问
- 支持错误提示和超时提示
- 支持长连接断线重连和单活消费

### 可选但建议在 MVP 内留接口

- 允许用户切换 repo 或 workspace
- 允许用户切换 agent 或 model
- 把执行摘要和 diff 摘要压缩展示到卡片中

## 用户场景

### 场景 1: 直接提问

用户在私聊里发一句“帮我检查这个仓库里的登录超时问题”，系统自动路由到默认仓库目录，创建或复用会话，异步执行后回推结果。

### 场景 2: 连续追问

用户在同一飞书线程中继续发“顺手把测试也补上”，系统命中原 sessionID，继续把消息追加到原会话。

### 场景 3: 权限审批

OpenCode 需要执行高风险操作时触发 `permission.asked`，IMUI 发送飞书卡片，用户直接回复“1/2/3”或对应文本完成审批。

### 场景 4: 补充信息

OpenCode 需要确认方案、选择路径或补全参数时触发 `question.asked`，IMUI 发送卡片并把用户序号选择或文本回答回传给 OpenCode。

### 场景 5: 无公网入口部署

服务部署在办公室网络、家庭网络或公司 NAT 网络中，没有公网入口。进程主动与飞书建立长连接，仍然可以接收飞书消息事件和卡片动作，再通过 OpenCode 和飞书 API 完成整轮交互。

## 接入约束

- 飞书侧必须使用企业自建应用
- 飞书消息和卡片交互都走长连接模式
- 长连接收到事件后需要快速确认，重逻辑异步化
- 多实例部署时要避免同一应用多连接随机分流导致状态混乱

## 成功标准

- 用户无需打开终端即可在飞书完成一轮完整的 OpenCode 交互
- 一条消息从飞书进入到 OpenCode 回答完成的成功率达到可接受水平
- 审批和回问可以完整闭环，不需要人工介入服务端数据库修复
- 主要错误可被定位到飞书入口层、网关层或 OpenCode 层
- 在没有公网 IP 和公网域名的前提下，系统仍可稳定运行

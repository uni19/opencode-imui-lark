# 13 Feishu Scope Minimum

## 目标

这份文档用于把飞书权限收敛到“满足当前功能所需的最小集合”，避免为了联调方便一次性开太多 scope。

注意：

- 飞书控制台里的 scope 名称可能会随版本调整
- 发布前应以控制台当前展示为准
- 本文档给的是“能力清单 + 最小化原则”，不是一份硬编码的控制台截图

## 1. 当前功能真正依赖的能力

围绕当前 IMUI 的功能，最小能力可以拆成 4 组。

### A. 接收消息与事件

用于：

- 订阅 `im.message.receive_v1`
- 收到私聊、群聊、thread 内的文本 / 图片 / 文件 / `post` 富文本

最小化原则：

- 只开“接收消息事件”所需能力
- 不开与通讯录、日历、审批、云文档无关的 scope

### B. 发送、回复和更新消息

用于：

- 发送文本消息
- 回复线程消息
- 发送共享卡片
- 更新处理中卡片、审批卡片、问题卡片和最终结果卡片

最小化原则：

- 只开消息发送、回复、更新所需能力
- 如果当前版本不需要主动建群、拉人、改群资料，就不要开群管理类 scope

### C. 下载消息附件

用于：

- 下载图片
- 下载文件
- 处理 `post` 富文本里的资源引用

最小化原则：

- 只开消息资源读取能力
- 不额外申请云空间文件管理、知识库、文档编辑等能力

### D. 长连接与应用身份

用于：

- 通过长连接接收事件
- 获取 tenant/app 访问令牌
- 识别当前应用和 bot 身份

最小化原则：

- 只保留应用基础身份、长连接和访问令牌能力
- 不因为未来可能支持别的产品能力，就提前申请额外权限

## 2. 功能到 scope 的核对表

发布前按下面表逐项确认。

| 功能 | 需要的能力组 | 如果不需要，就不要开的范围 |
| --- | --- | --- |
| 私聊 / 群聊接收消息 | A | 通讯录、日历、文档、审批、邮件 |
| 回复文本和卡片 | B | 群管理、用户管理、机器人以外的 IM 能力 |
| 卡片 patch / 共享卡片 | B | 不相关的应用 UI 或工作台权限 |
| 下载图片 / 文件 | C | 云盘文件管理、知识库节点管理 |
| 长连接事件订阅 | D | 与事件订阅无关的产品权限 |

## 3. 发布前人工确认方法

建议按下面顺序检查：

1. 打开飞书开放平台控制台
2. 仅保留 A/B/C/D 四组能力相关 scope
3. 对照 [README.md](../README.md) 和 [docs/10-release-checklist.md](10-release-checklist.md) 中的真实功能逐项核对
4. 在联调群和私聊中完成一轮最小回归
5. 如果某条功能正常，但 scope 明显超出上面四组，继续收缩

## 4. 最小回归集

scope 调整后，至少跑一轮：

- 私聊文本问答
- 群聊首条 `@bot`
- 发送共享卡片与卡片更新
- 图片上传 + 文本说明
- 文件上传 + 文本说明

如果其中任何一项失败，再回头定位对应能力组：

- 收不到消息：看 A
- 发不出消息 / patch 失败：看 B
- 图 / 文件下载失败：看 C
- 长连接起不来 / token 失败：看 D

## 5. 不建议默认打开的能力

当前版本没有直接依赖下面这些高风险或高范围能力，默认不建议申请：

- 通讯录读写
- 日历读写
- 云文档 / 云盘管理
- 多维表格
- 审批流管理
- 邮件
- 视频会议
- 知识库节点管理

如果后续真的要加这些能力，应该：

1. 先在需求文档中声明新增能力
2. 再更新本 scope 文档
3. 最后补自动化 / 手工回归

## 6. 当前结论

对这个项目来说，scope 最小化不是“把功能砍掉”，而是：

- 只围绕 IM 接收
- 只围绕消息发送与卡片更新
- 只围绕消息附件下载
- 只围绕长连接和应用身份

超出这四类的权限，都应该默认视为“待证明必要性”。
